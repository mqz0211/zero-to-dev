# Contributing

This repo is only as good as it is accurate and current. Contributions
of any size are welcome.

## Ways to contribute

### 1. Fix a broken link or outdated resource
Smallest, fastest contribution. Open a PR updating or removing it, or
open an issue if you're not sure how to fix it.

### 2. Suggest a resource (course, book, video, site)
Use the [resource suggestion issue template](.github/ISSUE_TEMPLATE/resource-suggestion.md).
The bar for inclusion: it should be **free** (or clearly marked if not),
**currently available**, and **teach something the existing list doesn't
already cover**. Curated beats comprehensive — we're not trying to list
every course that exists, only the best ones per topic.

### 3. Add a project idea
Add it to the right difficulty file in [`PROJECTS/`](PROJECTS/),
matching the existing format: a one-line description, the skills it
teaches, and what makes it more interesting than the most obvious
version of the idea.

### 4. Improve a roadmap
Roadmaps in [`ROADMAPS/`](ROADMAPS/) should stay sequenced (later
stages build on earlier ones) and scoped (a student should be able to
realistically follow it, not feel overwhelmed). If you want to add a
stage or reorder something, explain your reasoning in the PR.

### 5. Add a new fundamentals topic
[`FUNDAMENTALS/`](FUNDAMENTALS/) exists specifically for concepts other
roadmaps assume you already know. If you found yourself confused by a
term used elsewhere in this repo, that confusion is valuable — write
the explanation you wish you'd had, in plain language, no assumed prior
knowledge.

## Style guidelines
- Write for someone with **zero background** in the topic, even in
  intermediate/advanced files — define terms on first use
- Prefer short paragraphs and lists over dense prose — this repo is
  meant to be skimmed and referenced, not read cover to cover
- Every roadmap stage should answer: **what to learn, why it matters,
  and what to build with it**
- Keep tone plain and direct — no filler, no unearned hype

## Submitting a change
1. Fork the repo
2. Create a branch (`git checkout -b add-cloud-security-resources`)
3. Make your change
4. Open a PR with a short description of what changed and why

## Good first issues
Issues tagged [`good first issue`](../../labels/good%20first%20issue)
are scoped, well-defined tasks — a good place to make your first
contribution to this or any open-source project.

## Questions
Open an issue — there's no such thing as a question too basic for a
repo whose entire purpose is explaining the basics.
