# Intermediate Projects

> Assumes you've finished at least one full [beginner track](beginner-projects.md).
> These projects introduce real-world constraints: persistence, users,
> deployment, and dealing with messier data.

##  Python / Backend track

```
Project 1: REST API for a Notes App (Flask/FastAPI)
   ↓
Project 2: Add a real database (SQLite → PostgreSQL)
   ↓
Project 3: Add user authentication (login/signup)
   ↓
Project 4: Deploy it live (real users can access it)
```

| # | Project | New skill introduced |
|---|---|---|
| 1 | Notes App REST API | Flask/FastAPI, routing, JSON responses |
| 2 | Add a real database | SQL basics, ORMs (SQLAlchemy), migrations |
| 3 | Add authentication | Password hashing, sessions/JWTs |
| 4 | Deploy it | Environment variables, a hosting platform, basic DevOps |

##  Cybersecurity track

```
Project 1: Vulnerable Web App (build one, then break it yourself)
   ↓
Project 2: Log Analysis Tool (parse and flag suspicious activity)
   ↓
Project 3: Network Traffic Analyzer (packet inspection basics)
   ↓
Project 4: Automated Vulnerability Scanner (against your own lab apps)
```

| # | Project | New skill introduced |
|---|---|---|
| 1 | Vulnerable Web App | Understanding OWASP Top 10 from the builder's side |
| 2 | Log Analysis Tool | Regex, pattern detection, basic anomaly flagging |
| 3 | Network Traffic Analyzer | Packet capture basics (Scapy/Wireshark) |
| 4 | Vulnerability Scanner | Automating checks against your own local lab — **never external targets without permission** |

## What "intermediate" actually means here

These projects introduce the things tutorials often skip: real
databases instead of in-memory lists, actual users instead of a single
hardcoded one, and deployment instead of "runs on my machine." This is
the gap between "I followed a tutorial" and "I can build something
real" — and it's usually the gap employers/internship reviewers are
actually screening for.
