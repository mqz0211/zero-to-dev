# Building a Portfolio

## The core idea
A portfolio's job is to let someone assess your skill in under 5
minutes without talking to you. Every choice below serves that goal.

## How many projects
**2–4 real projects beats 10 tutorial clones.** A recruiter or hiring
manager will look closely at one or two projects, not skim through ten.
Depth over quantity, every time.

## What makes a project "real" vs. a "tutorial clone"
| Tutorial clone | Real project |
|---|---|
| Followed a video step by step | Started from your own idea or a genuine problem |
| Same features as everyone else who did the same tutorial | Has at least one feature/decision that's yours |
| No tests, no error handling | Handles edge cases, has at least basic tests |
| README is just the tutorial's default | README explains *your* decisions |
| Never deployed | Live and usable (if it's a web/app project) |

Following a tutorial to learn is completely fine — just don't stop
there. Extend it, break it on purpose and fix it, or rebuild one part
your own way.

## Each project needs
1. **A clear, well-written README** — see [`github-guide.md`](github-guide.md)
   for exactly what this should contain
2. **Clean commit history** — meaningful commit messages, not
   `fix`, `fix2`, `final fix`
3. **A live demo link**, if applicable — friction kills interest; nobody
   wants to clone and run your code just to see what it does
4. **Comments where they add real value** — not on every line, but on
   the non-obvious decisions

## Where your portfolio lives
- **GitHub** — this is your primary portfolio; pin your best 4–6 repos
- **A personal site** (optional but valuable, especially for web/design
  roles) — even a single page listing your projects with links works
- **A resume** — a distilled, one-page version pointing to both of the above

## Common mistakes
- **Too many unfinished projects** — a finished small project beats an
  ambitious unfinished one, every time
- **No context in the README** — someone landing on your repo cold
  should understand what it does in 10 seconds
- **Copy-pasted project ideas with zero personal twist** — if a
  recruiter has seen the exact same to-do app 50 times, yours won't stand out
- **Ignoring the non-code parts** — a great project with a terrible
  README performs worse than a good project with a great one

## Keeping it current
Revisit your pinned projects every few months. Delete or archive ones
that no longer represent your best work — a portfolio is a living
document, not a permanent record of everything you've ever built.
